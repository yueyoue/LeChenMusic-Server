# LeChenMusic 全栈开发文档（服务端重构 + 客户端适配）

## 1. 项目概述

### 1.1 项目背景与目标

基于现有 LeChenMusic Android 客户端项目，自研完整的音乐流媒体服务端与后台管理系统，替代 Navidrome 服务端，实现「私有部署 + 混合存储 + 多端适配」的个人/家庭音乐流媒体方案。

- **部署形态**：支持 NAS 与云服务器 Docker 容器化部署
- **存储能力**：统一兼容 NAS 本地磁盘、SMB/NFS 挂载目录；云盘通过 rclone 挂载后统一走本地驱动
- **功能基准**：核心功能参考道理鱼音乐，按需裁剪定制
- **客户端**：在现有 Kotlin Android 项目基础上改造，兼容手机、平板多尺寸屏幕

### 1.2 范围界定

- **包含**：服务端业务逻辑、RESTful API、Subsonic 兼容 API、后台管理系统、Android 客户端适配、Docker 部署方案
- **暂不包含**：iOS 客户端、Web 播放端、社交分享、付费会员体系

### 1.3 与现有项目的关系

- **APP 端**：从原 LeChenMusic 项目复制代码，在此基础上改造网络层和 UI，渐进式迁移
- **服务端**：完全重写，不依赖 Navidrome，同时提供 Subsonic 兼容 API 以便现有客户端平滑过渡

## 2. 整体架构设计

### 2.1 系统分层架构

采用前后端分离 + 分层解耦设计，从上到下分为 5 层：

```
┌─────────────────────────────────────────────────────────┐
│  接入层：Nginx 反向代理（静态资源 / HTTPS / 跨域 / 转发）  │
├─────────────────────────────────────────────────────────┤
│  应用层：Node.js 业务服务                                 │
│  ┌───────────┬──────────┬───────────┬──────────────────┐ │
│  │ REST API  │ Subsonic │ 管理后台   │ 转码服务 / 扫描   │ │
│  │   接口    │ 兼容 API │  静态资源  │   定时任务        │ │
│  └───────────┴──────────┴───────────┴──────────────────┘ │
├─────────────────────────────────────────────────────────┤
│  数据层：SQLite（默认）/ MySQL（可选）+ 可选 Redis 缓存    │
├─────────────────────────────────────────────────────────┤
│  存储抽象层：统一 StorageDriver 接口                      │
│  ┌──────────┬──────────┬─────────────────────────────┐  │
│  │ 本地磁盘  │ SMB/NFS  │ rclone 挂载（阿里云盘等）    │  │
│  └──────────┴──────────┴─────────────────────────────┘  │
├─────────────────────────────────────────────────────────┤
│  客户端层：Android 原生 APP（手机 / 平板）                 │
└─────────────────────────────────────────────────────────┘
```

### 2.2 核心业务流程

- **媒体入库**：管理员配置存储路径 → 后台触发扫描 → 解析音频元数据 → 写入数据库 → 生成封面缩略图
- **播放请求**：APP 发起播放 → 服务端鉴权 → 识别文件存储源 → 读取文件流 → 按需转码 → 流式返回客户端
- **管理操作**：后台登录 → 媒体库管理 / 用户管理 / 转码配置 / 存储配置 → 配置持久化生效

### 4.2 核心模块设计

#### 4.2.1 混合存储抽象层

核心设计：定义统一 `StorageDriver` 接口，所有存储源实现同一套读写方法，上层业务完全不感知底层存储位置。

```typescript
interface StorageDriver {
  // 判断文件是否存在
  exists(path: string): Promise<boolean>;
  // 获取文件可读流（核心：用于播放传输）
  createReadStream(path: string, range?: { start: number; end: number }): Promise<Readable>;
  // 获取文件元信息（大小、修改时间）
  stat(path: string): Promise<FileStat>;
  // 遍历目录（用于媒体库扫描）
  listDir(path: string): Promise<FileItem[]>;
  // 获取存储源类型标识
  getType(): StorageType;
}

type StorageType = 'local' | 'smb' | 'nfs';
```

- 内置驱动：本地磁盘、SMB/NFS 网络挂载
- 云盘方案：通过 rclone 挂载为本地目录，统一走本地驱动，不单独实现云盘 API
- 数据库中每条音轨记录 `storage_type` 与 `storage_path` 字段，播放时自动路由到对应驱动
- **路径安全**：所有文件路径必须经过规范化处理（`path.resolve` + 前缀校验），防止 `../../../` 路径遍历攻击

#### 4.2.2 实时转码模块

- **工作模式**：按需触发，默认直链播放，仅客户端指定码率或格式不兼容时启动转码
- **实现方式**：子进程调用 FFmpeg，输入流直接对接文件读取流，输出流 pipe 到 HTTP 响应，全程无临时文件
- **转码档位**：无损直传、320kbps MP3、256kbps AAC、128kbps AAC 四档
- **缓存策略**：转码完成的文件写入本地缓存目录，按 LRU 策略自动清理，相同请求直接返回缓存
- **并发限制**：可配置最大同时转码路数，超出则排队或降级直链，保护 NAS CPU 资源
- **进程管理**：
  - 客户端断开连接时，通过 AbortController 触发对应 FFmpeg 子进程 `process.kill()`
  - 转码进度通过 SSE（Server-Sent Events）推送给客户端
  - FFmpeg 不支持的格式（如 DSD）自动降级为直链播放，返回原始文件流
- **转码缓存表**：记录已转码文件的路径、码率、源文件哈希，命中缓存时直接返回，无需重新转码

#### 4.2.3 Subsonic 兼容 API

为实现现有客户端平滑迁移和第三方客户端兼容，在 `/rest/` 路径下实现 Subsonic API 协议：

- `ping`、`getMusicFolders`、`getArtists`、`getAlbum`、`getSong`
- `stream`（播放）、`search2`、`getPlaylists`、`createPlaylist`
- `scrobble`（播放上报）、`getAlbumList2`、`getArtistInfo2`

认证方式：用户名 + 密码 MD5/token，与自研 JWT 鉴权独立运行。

### 4.3 鉴权与安全

#### 4.3.1 JWT 鉴权方案

- **Access Token**：有效期 2 小时，存储在客户端内存
- **Refresh Token**：有效期 30 天，存储在客户端安全存储，用于无感续期
- **密码哈希**：使用 argon2id 算法
- **多设备策略**：允许多设备同时登录，每个设备独立 Token，支持单独踢出

#### 4.3.2 安全加固

- HTTPS 强制（Nginx 层配置）
- 请求频率限制（express-rate-limit），防止暴力破解
- 文件路径防遍历（规范化路径 + 白名单前缀校验）
- CORS 白名单配置
- 管理后台接口额外的权限校验中间件

### 4.5 API 接口规范

- 统一采用 RESTful 风格
- 响应格式：`{ code: number, message: string, data: any }`
- 鉴权方式：JWT Token（Bearer），接口按角色权限拦截
- API 文档：Swagger/OpenAPI 自动生成，访问 `/api-docs`
- 错误码规范：
  - `0` 成功
  - `400xx` 参数错误
  - `401xx` 认证失败
  - `403xx` 权限不足
  - `404xx` 资源不存在
  - `500xx` 服务端错误

核心接口分组：

| 分组 | 主要接口 |
|------|---------|
| 认证 | POST /auth/login, POST /auth/register, POST /auth/refresh, POST /auth/logout |
| 媒体库 | GET /api/libraries, POST /api/libraries, POST /api/libraries/:id/scan |
| 艺人 | GET /api/artists, GET /api/artists/:id, GET /api/artists/:id/albums |
| 专辑 | GET /api/albums, GET /api/albums/:id, GET /api/albums/:id/tracks |
| 音轨 | GET /api/tracks, GET /api/tracks/:id, GET /api/tracks/:id/stream, GET /api/tracks/:id/lyrics |
| 搜索 | GET /api/search?q=xxx |
| 歌单 | GET /api/playlists, POST /api/playlists, PUT /api/playlists/:id, DELETE /api/playlists/:id |
| 收藏 | GET /api/favorites, POST /api/favorites, DELETE /api/favorites/:type/:id |
| 播放历史 | GET /api/history, POST /api/history |
| 后台管理 | /admin/* （用户管理、系统配置、转码设置、存储管理、缓存管理） |
| Subsonic | /rest/* （Subsonic API 兼容接口） |

## 5. Android 客户端改造方案

### 5.1 改造原则

- 最大程度复用现有项目的 UI 组件、播放逻辑与交互体验
- 仅替换数据源层与适配新接口，不推翻原有架构
- 增量增加平板适配、码率切换、多端同步等新特性
- **渐进式迁移**：先通过 Subsonic 兼容 API 保持现有功能可用，再逐步切换到自研 API

### 5.2 核心改造点

#### 网络层替换

- 保留现有 Subsonic API 对接代码，使其连接到自研服务端的 Subsonic 兼容接口
- 新增自研 REST API 的 Retrofit 接口定义
- 统一封装请求拦截器，处理 Token 鉴权、错误码统一处理、Token 自动刷新

#### 播放内核优化

- 保留 ExoPlayer 播放核心
- 新增「直链播放」与「转码播放」两种模式自动切换
- 增加 10~30 秒预缓冲策略，优化弱网与 WiFi 波动场景的播放稳定性
- 新增播放进度上报，实现跨设备续播

#### 数据同步适配

- 歌单、收藏、播放历史改为与服务端双向同步
- 新增离线缓存功能，支持缓存指定码率的音频到本地

#### 设置中心扩展

- 新增服务端地址配置、账号登录入口
- 新增播放码率选择（自动 / 无损 / 320k / 128k）
- 新增「内网自动无损、外网自动降码」的智能切换选项

### 5.3 多屏幕适配方案（手机 + 平板）

#### 布局分级适配

- 手机端（<600dp）：沿用现有单栏竖向布局，底部播放栏 + 页面切换
- 平板端（≥600dp）：采用双栏布局，左侧常驻导航/歌单列表，右侧展示详情内容与播放控制

#### 播放页适配

- 平板端横屏模式优化：左侧专辑封面 + 歌词，右侧播放控制与歌单
- 字体、间距随屏幕尺寸动态缩放

#### 技术实现

- 使用 `sw<N>dp` 限定符区分屏幕宽度
- 公共组件封装为自适应组件

## 7. 部署与运维方案

### 7.1 Docker 容器化部署

采用单镜像打包，内置 Node.js、FFmpeg、管理端静态资源，支持 SQLite 开箱即用。

```yaml
# docker-compose.yml
version: '3'
services:
  lechen-music:
    image: lechen-music:latest
    container_name: lechen-music
    restart: unless-stopped
    ports:
      - "3000:3000"  # API + 管理后台端口
    environment:
      - TZ=Asia/Shanghai
      - DB_TYPE=sqlite
      - MAX_TRANSCODE_TASKS=4
    volumes:
      - ./data:/app/data           # 数据库、配置、转码缓存
      - /path/to/music:/music:ro   # 本地音乐目录挂载（只读）
    deploy:
      resources:
        limits:
          cpus: '2.0'
          memory: 2G
```

### 7.2 NAS 部署适配

- **权限适配**：容器内运行用户与 NAS 宿主用户 UID/GID 对齐
- **资源限制**：默认限制 2 核 CPU + 2G 内存
- **存储挂载**：支持本地路径挂载、SMB 目录挂载两种方式
- **架构支持**：同时构建 x86_64、ARM64 镜像

### 7.3 云服务器部署适配

- 可选 MySQL + Redis 架构，提升多用户并发性能
- 配合 rclone 挂载云盘音源，统一走本地驱动

### 7.4 日志规范

- 结构化 JSON 日志，方便 ELK/Loki 等工具采集
- 日志级别：error / warn / info / debug
- 日志轮转：按大小（50MB）和天数（30天）自动轮转
- 播放请求单独记录访问日志

### 7.5 备份策略

- SQLite 数据库文件支持定时备份（crontab）
- 提供数据导出/导入 API
- 提供从 Navidrome 迁移的命令行工具（导入用户、播放列表、收藏）

## 9. 性能与稳定性优化建议

- **媒体扫描优化**：增量扫描基于文件修改时间，避免重复解析；批量入库采用事务写入
- **转码优化**：常用码率优先缓存，冷门歌曲按需转码；APE 格式扫描时预转 FLAC 缓存
- **客户端优化**：列表图片三级缓存、预加载下一首音频
- **资源隔离**：Docker 严格限制 CPU/内存配额
- **进程管理**：FFmpeg 子进程超时自动清理，避免僵尸进程

## 10. 错误处理规范

### 服务端错误码体系

| 范围 | 含义 |
|------|------|
| 0 | 成功 |
| 1001-1999 | 认证错误（Token 过期、密码错误、账号锁定） |
| 2001-2999 | 参数错误（缺少必填项、格式无效） |
| 3001-3999 | 资源不存在（歌曲、专辑、用户未找到） |
| 4001-4999 | 权限不足（非管理员访问管理接口） |
| 5001-5999 | 业务错误（扫描进行中、转码队列满） |
| 9001-9999 | 服务端内部错误 |

### 客户端错误重试策略

- 网络超时：自动重试 3 次，指数退避（1s/2s/4s）
- Token 过期：自动用 Refresh Token 续期，失败则跳转登录
- 播放失败：自动降级码率重试（无损 → 320k → 128k）
- 扫描/转码失败：记录错误日志，通知管理员
